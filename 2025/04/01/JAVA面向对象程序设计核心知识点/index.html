JAVA面向对象程序设计核心知识点

一、类与对象

定义

  • 类：对象的模板，定义属性和方法（如public class Car { private String brand; }）。

  • 对象

    ：类的实例，具有状态（字段值）和行为（方法调用）。例如：

关键特性

  • 封装性：通过访问限定符隐藏内部实现。
  • 继承性：子类通过extends复用父类代码。
  • 多态性：方法重写（Override）和重载（Overload）。

二、访问限定符的意义

限定符 作用范围
public 所有类均可访问
protected 同包或子类可访问
默认 仅同包内可访问
private 仅类内部可访问

级别的高低

private>default>protected>public。

设计原则：

  • 默认使用private，按需开放权限。
  • 通过getter/setter控制字段访问。

三、封装

封装是Java面向对象编程的三大核心特性之一，它通过隐藏对象内部状态和实现细节，仅对外暴露必要接口，从而提高代码的安全性、可维护性和扩展性。

封装的核心作用

数据隐藏与保护
通过将字段设为private，外部无法直接访问或修改内部数据，必须通过受控的公共方法（如getter/setter）操作。

通过这样的操作，其他的类就无法直接访问对象的内部信息，而是通过类提供的方法实现对内部信息的访问。

五，构造方法

1，定义构造方法

构造方法是一个特殊的成员方法，有以下几点需要注意。

（1）构造方法的名称必须与类名一致。

（2）构造方法没有任何返回值，前面不可以加任何返回值的类型连void都不行。

（3）由于没有返回值，所以构造方法没有return。

2，构造方法的实际应用

构造方法分为有参构造和无参构造

六、Bean的格式约束

JavaBean是Java编程中一种特殊的类规范，主要用于封装数据和提供标准访问方式。

标准的javabean类应符合以下要求

1，类名需要舰名思意

2，成员需要用private修饰

3，提供至少两个构造方法

（1）无参构造方法

（2）带全部参数的构造方法

4，成员方法

​ (1)提供每一个成员变量对应的setXxx（）/getXxx（）；注意：这两个函数必须依照严格的命名格式！

​ (2)如果还有其他行为，也需要写上。

📌 三大核心规范
  1. 私有字段
    所有属性必须用private修饰，禁止外部直接访问（保证数据安全性）
  2. 公共无参构造方法
    必须提供无参数的public构造方法（方便反射机制创建对象）
  3. 公共的getter/setter方法
    为每个属性提供getXxx()和setXxx()方法（用于安全访问和修改数据）
🌰 典型示例
  1. 数据传输对象（DTO）
    在层与层之间传递数据（如Controller层→Service层）
  2. 模型层（Model）
    在MVC模式中作为数据模型存储业务数据
  3. 配置信息存储
    通过properties文件配置Bean属性（结合Spring框架）
  4. ORM映射
    与数据库表字段建立对应关系（如MyBatis/Hibernate）
🔍 优势特点
  • 封装性：隐藏内部实现细节，通过方法控制访问
  • 可重用性：标准化设计便于跨系统使用
  • 工具支持：IDE可自动生成getter/setter方法

八，继承

继承是Java面向对象编程（OOP）的三大核心特性之一（继承、封装、多态），它允许一个类（子类）继承另一个类（父类）的属性和方法，从而实现代码复用和扩展。以下是详细解释：

📌 核心概念

继承关系：通过extends关键字建立
成员访问规则：

  • public：子类可直接访问
  • protected：子类可直接访问
  • 默认(包)：仅当子类与父类同包时可访问
  • private：子类不可直接访问
🚀 继承优势
  1. 代码复用：避免重复编写相同代码
  2. 扩展性：在父类基础上添加新特性
  3. 多态基础：支持方法重写和父类引用指向子类对象
📝 语法实现
🔍 继承类型

  1. 单继承：Java只支持单继承（一个子类只能有一个直接父类）

  2. 多级继承：形成继承链

    注意：当父类中的内容被子类更改以后，子类的子类再次调用时是更改以后的内容

⚠️ 注意事项

  1. 构造方法调用：子类构造方法默认调用父类无参构造方法（可用super()显式调用）

  2. 方法重写规则

    ：

    • 方法名、参数列表必须与父类方法完全一致
    • 返回类型可以是父类方法返回类型的子类
    • 访问权限不能比父类方法更严格

  3. 合理使用继承：优先考虑组合（has-a关系）而非继承（is-a关系），避免过度继承导致类层次复杂
